담당 영역

매일 트렌딩 영상 메타데이터를 수집해 GCS에 적재. ResilientYouTubeClient가 API 호출의 신뢰성을 담당.

해결한 문제

단일 Key → 무효화 시 전체 중단. 부분 실패 허용 없음. IP밴 대응 부재. 점진적 복구 + 운영 알람 최적화로 해결.

복원 철학

API 호출 실패 시 점진적 강도의 4단계로 복구. 각 단계는 서로 다른 실패 유형을 흡분해 부분 실패를 허용합니다.

핵심 설계 결정

accessNotConfigured=프로젝트 스코프(회전 무효) · userRateLimitExceeded=보수적 회전 무효 · IP밴 시그니처는 최소 Key≥2일 때만 판정.

보안 불변량

로깅은 key_index만 허용. Key값·헤더·본문·traceback·URL 전체 금지. from None 체인 scrub으로 credentials 누출 차단.

_classify_error()는 순수 함수로 status_code + reason → Verdict를 결정. 테스트 용이하며 외부 상태 의존성 없음.

분류 기준

응답의 status_code와 reason을 5종 Verdict로 매핑. 각 Verdict가 어느 복원 단계를 트리거할지 결정합니다.

설계 핵심

프로젝트 스코프 실패(quota·config)는 회전 무효 → 즉시 종료로 불필요한 재시도·알람 방지. IP밴은 스코프가 다르므로 별도 단계.

운영 효과

쿼터 소진(예상 종료)과 진짜 장애를 분리 → 알람 피로 감소. Key 무효화는 자동 회전으로 무중단.

egress IP 회전 — 두 경로
Normal
Client→ 직접 호출→ egress IP₁→ YouTube API
Proxy
Client⇢ Cloud Run→ dumb forwarder→ egress IP₂→ YouTube API

IP밴 시그니처 감지(전 활성 Key 동일 403) 시 Normal → Proxy로 자동 전환. Cloud Run revision 재배포 시 새 egress IP 할당.

프록시 역할

dumb forwarder — 비즈니스 로직 없이 egress IP만 전환. host 화이트리스트(www.googleapis.com)로 목적지 제한.

보안 설계

path escape 차단 · key= query 거부 · X-Goog-Api-Key 헤더 필수 · once-True-stays-True unhealthy(Cloud Run 재시작 유도).

ADR 0001

프록시는 학습+범용 egress seam 목적. IP밴은 꼬리 케이스. 3차 배포(Terraform)는 인프라 담당자 영역으로 보류.

IP밴 시그니처

3가지 핵심 발견

1. quota 폭주로 IP밴 유도 불가 — YouTube는 quota 초과 시 quotaExceeded만 반환 (별개 메커니즘)
2. quota 초과 시 복원력 정상 동작 — TERMINAL_QUOTA → 즉시 CollectionExhausted
3. 정상 사용(7 units/day)에서 IP밴 미발생 — ADR 0001 가설 실증
