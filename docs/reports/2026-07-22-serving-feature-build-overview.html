PR #220 · Autoresearch Serving

Serving Feature Build

추천을 요청하는 쪽은 이제 사용자와 영상 ID만 보냅니다. 서버가 필요한 정보를 직접 모아 CTR 점수를 계산하는 흐름으로 바뀌었습니다.

코드·CI 수준 구현됨 운영 전제 다른 저장소 소유

핵심 변화

호출자가 피처를 만들던 방식에서,
서버가 일관된 피처를 조립하는 방식으로 책임이 옮겨갔습니다.

ID만요청에는 user_id와 video_ids만
2회Feast 배치 조회로 필요한 정보 결합
15개학습과 맞춘 모델 입력 피처
503준비되지 않은 상태를 숨기지 않음

책임의 이동

변경 전 · 후

핵심은 API를 작게 만든 것이 아니라, 추천 점수의 근거가 호출자마다 달라지지 않게 한 것입니다.

변경 전

호출자가 피처를 만들어 전달

영상별 정보와 사용자 정보를 요청에 담아야 했습니다. 호출자마다 계산 방식이 달라지면 학습 때 사용한 기준과 실제 점수 기준이 어긋날 수 있었습니다.

현재

호출자는 ID만, 서버는 피처를 직접 조립

서버가 온라인 피처 저장소에서 같은 기준의 정보를 꺼내고 모델 입력을 만듭니다. 호출자는 피처의 세부 구조를 알 필요가 없습니다.

한 번의 추천 요청 안에서

새 요청 흐름

영상은 최대 200개까지 한 번에 보낼 수 있습니다. 서버는 같은 요청 안에서 필요한 정보를 묶어 조회하고, 각 영상의 CTR 점수를 돌려줍니다.

사용자와 영상 ID를 보냄

/rerank에는 user_id와 video_ids만 들어옵니다.

호출자의 부담 감소
첫 번째로 기본 정보를 모음

Feast에서 사용자·영상 관련 정보를 한 번에 읽습니다.

UserStatic · UserDynamic · Video
두 번째로 취향 유사도를 더함

영상 카테고리별로 사용자의 관심사와 얼마나 맞는지 읽어 결합합니다.

UserCategorySimilarity
15개 신호로 CTR 계산

학습에 사용한 순서와 타입에 맞춰 모델 입력을 만들고, 영상별 점수를 반환합니다.

응답은 요청한 영상 순서를 유지
중요: 응답은 CTR 점수를 포함하지만, 영상의 나열 순서는 요청한 video_ids 순서를 보존합니다. 화면에서 순위를 바꾸고 싶다면 호출자가 점수를 기준으로 정렬할 수 있습니다.

프로젝트 전체에 생긴 효과

추천의 기준을 한곳에 모았습니다

01

호출이 단순해집니다

클라이언트와 다른 서비스는 피처 표를 만들 필요 없이 사용자·영상 ID만 보냅니다.

02

학습과 서빙의 기준이 가까워집니다

모델에 넣는 15개 신호의 순서·타입을 계약으로 고정해, 점수 근거의 일관성을 높입니다.

03

준비 안 된 상태를 숨기지 않습니다

모델, 온라인 피처 저장소, 피처 계약 중 하나라도 준비되지 않으면 정상처럼 답하지 않고 503으로 드러냅니다.

04

관측할 수 있습니다

요청 길이·응답 시간·준비 상태와 처음 보는 카테고리 값을 메트릭으로 남겨 재학습이 필요한 신호를 볼 수 있습니다.

함께 움직이지만, 같은 책임은 아닙니다
